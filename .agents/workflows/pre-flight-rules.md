# 🛡️ GOPEAK ERP — กฎข้อบังคับสำหรับ AI Agent (Mandatory Pre-Flight Rules)

กฎเหล่านี้ **บังคับใช้ทุกครั้ง** ก่อนสร้าง แก้ไข หรือลบไฟล์ใดๆ ในโปรเจค ห้ามข้ามขั้นตอนใดเด็ดขาด

---

## กฎที่ 1: สำรวจโฟลเดอร์ก่อนเขียนโค้ดเสมอ (Mandatory Folder Recon)

**ก่อนสร้างหรือแก้ไขไฟล์ใดๆ** ต้องทำทุกข้อดังนี้:

1. ใช้ `list_dir` หรือ `find_by_name` สแกนโฟลเดอร์ปลายทางและโฟลเดอร์ใกล้เคียง เพื่อทำความเข้าใจโครงสร้างไฟล์ที่มีอยู่แล้ว
2. ถ้าไฟล์เป้าหมายมีอยู่แล้ว → ใช้ `view_file` อ่านเนื้อหาเดิมทั้งหมดก่อน
3. ใช้ `git log --oneline -5 -- <file>` ตรวจสอบประวัติการแก้ไขของไฟล์นั้น

**ห้าม** สร้างไฟล์ใหม่ (`write_to_file`) โดยไม่ผ่านขั้นตอนข้างต้น

---

## กฎที่ 2: ห้ามเขียนทับไฟล์ที่มีอยู่แล้ว (No Overwrite Without Approval)

1. ถ้าไฟล์เป้าหมายมีอยู่แล้ว **ห้ามใช้ `write_to_file` กับ `Overwrite: true`** โดยเด็ดขาด
2. ให้ใช้ `replace_file_content` หรือ `multi_replace_file_content` เพื่อแก้ไขเฉพาะส่วนที่ต้องการแทน
3. หากจำเป็นต้องเขียนไฟล์ใหม่ทั้งไฟล์ (เช่น Refactor ครั้งใหญ่) → **ต้องแจ้งผู้ใช้ก่อนทุกครั้ง** ระบุว่า:
   - ไฟล์ไหนจะถูกเขียนทับ
   - เนื้อหาเดิมมีอะไรบ้าง (สรุปสั้นๆ)
   - ของใหม่จะต่างจากของเก่าอย่างไร
4. **รอการอนุมัติจากผู้ใช้** ก่อนดำเนินการ

---

## กฎที่ 3: Implementation Plan ต้องแจกแจงไฟล์ที่กระทบ (Impact Declaration)

ใน Implementation Plan ทุกฉบับ ต้องระบุให้ชัดเจนว่า:

- ไฟล์ไหนเป็น **[NEW]** (สร้างใหม่ — ยังไม่มีในระบบ)
- ไฟล์ไหนเป็น **[MODIFY]** (แก้ไขจากของเดิม — ระบุว่าแก้ส่วนไหน)
- ไฟล์ไหนเป็น **[OVERWRITE]** (เขียนทับทั้งไฟล์ — ต้องขอ Approval ก่อน)
- ไฟล์ไหนเป็น **[DELETE]** (ลบ — ต้องขอ Approval ก่อน)

ถ้าเป็น [MODIFY] ต้องอธิบายว่าจะแก้ "ส่วนไหน" ของไฟล์เดิม ไม่ใช่แค่บอกชื่อไฟล์

---

## กฎที่ 4: Git Commit ย่อยและบ่อย (Atomic Commits)

1. Commit ทุกครั้งหลังแก้ไขแต่ละ Feature สำเร็จ (ไม่ใช่รวมทุกอย่างเป็น commit เดียวตอนจบ)
2. Commit message ต้องชัดเจน ระบุ scope เช่น `feat(ess):`, `fix(hr):`, `refactor(sidebar):`
3. ก่อน Push ให้รัน `npm run build` เพื่อตรวจสอบ TypeScript errors ทุกครั้ง

---

## กฎที่ 5: ตรวจสอบ Dependencies ก่อนแก้ไข Shared Components

สำหรับไฟล์ที่ใช้ร่วมกันหลายหน้า เช่น `Sidebar.tsx`, `layout.tsx`, `auth.ts`, `schema.prisma`:

1. ใช้ `grep_search` ค้นหาว่ามีไฟล์ไหนอ้างอิง (import) ไฟล์นั้นบ้าง
2. ประเมินผลกระทบก่อนแก้ไข
3. แจ้งผู้ใช้ถ้ามีความเสี่ยงที่จะกระทบหน้าจออื่น
